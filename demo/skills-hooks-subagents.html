Skills / Hooks / Subagents

エージェント開発における3つの拡張機構と、コンテキスト管理の重要性

LLMエージェントは毎回ステートレスに起動する。記憶も判断力も、すべてはその瞬間のコンテキストウィンドウに渡された情報から生まれる。このウィンドウは有限であり、何を入れて何を入れないかが、エージェントの品質とコストを決定する。

Skills・Hooks・Subagentsは、この有限なコンテキストを効率的に運用するための3つの拡張機構である。それぞれ発火の仕組み、実行主体、コンテキストへの影響が異なる。
SKILLS

知識の外部化 — 必要なときだけ読み込む

📖 何であるか

Skillsは外部に置かれたマニュアル（SKILL.mdファイル）を、エージェントが必要と判断したときに読み込む仕組み。読み込まれた知識はエージェント自身のコンテキストに追加され、そのガイドに従って作業を実行する。

実行主体はメインエージェント自身。知識は増えるが、プロセスは増えない。

⚡ 発火の仕組み

非決定論的・セマンティックマッチング。

ユーザーの入力に対して、各Skillのdescriptionフィールドとの意味的な類似度をLLMが判断し、関連するSkillを選択して読み込む。同じ入力でも文脈次第で違うSkillが選ばれることがある。

  • 「プレゼン作って」→ pptxスキルが発火
  • 「PDF読んで」→ pdf-readingスキルが発火
  • エージェントが自律的に「これ読むべき」と判断して発火

📐 コンテキストへの影響

SKILL.mdの内容がメインのコンテキストウィンドウに追加される。つまり読み込んだ分だけトークンを消費する。ただし、システムプロンプトに全知識を詰め込むのと違い、必要なときだけ読むのでトークン効率が良い。

HOOKS

確定的なチェックポイント — 条件を満たせば必ず発火する

🔗 何であるか

Hooksはプログラムの特定フェーズで必ず実行される処理。ReactのuseEffectがレンダー後に必ず走るように、エージェントのHooksも特定のイベント（ファイル保存時、コマンド実行前後など）にバインドされ、条件を満たせば100%発火する。

AIの判断を介さない。ルールベースの自動実行。

⚡ 発火の仕組み

決定論的・イベント駆動。

あらかじめ定義されたトリガー条件（タイミング・イベント）に合致したら、LLMの判断を経ずに自動で実行される。

  • ファイル保存時 → リンターを自動実行
  • コミット前 → テストを自動実行
  • 毎回のレンダー後 → 副作用処理を実行（React useEffect）

📐 コンテキストへの影響

Hookの実行結果はメインのコンテキストに反映される。ただし、Hook自体は通常軽量な処理（バリデーション、フォーマットチェックなど）なので、コンテキストへの負荷は最小限。

SUBAGENTS

仕事の委譲 — 別プロセスに丸投げして結果だけ受け取る

🧬 何であるか

Subagentは独立したコンテキストウィンドウを持つ別のエージェントプロセス。メインエージェントが「これは任せた方がいい」と判断したときに生成され、タスクを独立して実行し、結果だけ親に返す。

実行主体が別に生まれる。LangGraphでいうサブグラフの呼び出しに相当。

⚡ 発火の仕組み

非決定論的・エージェント判断。

メインエージェントが「このタスクは自分でやるより別のプロセスに委譲すべき」と推論して発火する。Skillsと同じくAIの自律的判断だが、テキストを読むのではなく、プロセスを生成する点が異なる。

  • 「5ファイル並列リファクタ」→ 5つのサブエージェントにdispatch
  • 「ドキュメント調査して」→ 調査サブエージェントに委譲
  • メインの作業と無関係なサイドタスク → サブに切り出す

📐 コンテキストへの影響

メインのコンテキストを汚さない。これが最大のメリット。サブエージェントの思考過程・ツール実行ログはすべてサブ側のコンテキストで完結し、メインには最終結果のみが返る。メインのコンテキストはクリーンに保たれる。

CONTEXT MANAGEMENT

なぜコンテキスト管理がすべてを決めるのか

🧠 LLMの根本的制約

LLMは毎回ステートレスで起動する。記憶があるように見えるのは、毎回のリクエストに会話履歴・システムプロンプト・メモリをまとめて渡しているから。LLM自体は何も覚えていない。

コンテキストウィンドウは有限（Claude: 200kトークン）で、超過分は切り落とされるか要約される。エラーは出ず、静かに劣化する。これが最も危険なポイント。

💰 コスト構造の理解

システムプロンプトは常駐コスト。100往復の会話なら、同じシステムプロンプトが100回課金される。ここに全知識を詰め込むのは、コスト的にも品質的にも悪手。

だからこそ、常駐させるもの（システムプロンプト）は最小限にし、状況に応じて読み込むもの（Skills）、別プロセスに逃がすもの（Subagents）を使い分ける設計が重要になる。

🎯 3機構による最適化戦略

システムプロンプト / CLAUDE.md — 人格・判断基準・基本ルールだけ。「憲法」。常に読み込まれるので簡潔に。

Skills — 専門知識・手順書。必要なときだけ読み込む「業務マニュアル棚」。常駐コストゼロ。

Hooks — 品質保証の自動チェック。コンテキストへの負荷最小で確実に実行される「業務フローのチェックポイント」。

Subagents — 独立タスクの委譲先。メインのコンテキストを汚さず、並列実行も可能な「別部署への業務委託」。

核心：コンテキストウィンドウは「作業机の広さ」。机に乗り切らない資料は、棚に置いて必要なときだけ出す（Skills）、別の机で作業してもらう（Subagents）、決まったタイミングで自動チェックを走らせる（Hooks）。この設計が、エージェントの品質・コスト・スケーラビリティのすべてを決める。

実務への適用

🔧 設計判断フロー

「この知識/処理をどこに置くか」を決めるとき、以下の順で考える：

  • 毎回必要か？ → Yes → システムプロンプト（ただし最小限に）
  • 特定条件で必ず実行すべきか？ → Yes → Hooks
  • 特定タスクのときだけ必要な知識か？ → Yes → Skills
  • 途中経過がメインに不要か？ → Yes → Subagents

⚠️ よくあるアンチパターン

  • CLAUDE.mdに全手順書を書く → Skillsに外出しすべき
  • メインセッションで寄り道タスクをやる → Subagentに委譲すべき
  • 品質チェックをLLMの判断に任せる → Hooksで確定的に実行すべき
  • 全Skillを毎回読み込む → 必要なものだけ選択的に読むべき
